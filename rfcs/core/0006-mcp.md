# Summary

This RFC defines how Flow exposes itself as a **Model Context Protocol (MCP) server**, so that any MCP-aware agent (Claude, GPT, Gemini, Cursor, Codex) can edit videos by calling Flow's tools and reading its resources. The MCP surface is the primary agent-facing API.

MCP is the only protocol Flow speaks natively. All other protocols (HTTP REST, gRPC, WebSocket) are convenience layers for human-facing tools and are out of scope for this RFC.

# Motivation

Flow's user is an AI agent. The agent needs:

- A **tool surface** for actions: probe media, apply an action, start a render, check status.
- A **resource surface** for state: the current timeline, the available effects, the project's media.
- A **prompt surface** for guidance: how to use Flow, what the conventions are, what good output looks like.

MCP is the standard protocol that satisfies all three. It is supported by every major LLM client. It is text-friendly (JSON-RPC 2.0 over stdio, HTTP, or WebSocket). It has well-defined semantics for tools, resources, and prompts.

Building a custom protocol would force every agent author to write a custom client. Building an MCP server lets Flow plug into the existing ecosystem of MCP clients for free.

# Goals

- Expose Flow as an MCP server with a complete tool, resource, and prompt surface.
- Support stdio and HTTP transports.
- Make every action in the action system (RFC-0003) callable as an MCP tool.
- Make the timeline (RFC-0004) inspectable as MCP resources.
- Provide prompts that guide agents through common Flow workflows.
- Validate MCP requests against the same schemas as the runtime API (one source of truth).
- Stream progress for long-running operations via MCP notifications.
- Be a good MCP citizen: respect the protocol's semantics, return well-typed errors.

# Non Goals

- This RFC does **not** define the non-MCP HTTP API (a convenience for `flow-server`'s web UI; not an agent surface).
- It does **not** redefine MCP. Flow speaks MCP as specified.
- It does **not** define how the agent interprets Flow's responses. That is the agent's job.
- It does **not** define authentication or authorization. MCP is bearer-token-agnostic; Flow's `flow-server` handles auth at a layer below MCP.

# Guide-level explanation

An MCP client (e.g. Claude) connects to `flow-server`'s MCP endpoint and discovers the available tools:

```
> tools/list
[
  { name: "flow.media.probe", description: "Probe a media file and return its metadata.", inputSchema: {...} },
  { name: "flow.project.list", description: "List all projects in the server.", inputSchema: {...} },
  { name: "flow.project.create", description: "Create a new project.", inputSchema: {...} },
  { name: "flow.timeline.get", description: "Get the current timeline as OTIO JSON.", inputSchema: {...} },
  { name: "flow.timeline.apply", description: "Apply an action or batch of actions.", inputSchema: {...} },
  { name: "flow.timeline.plan", description: "Validate an action without applying it.", inputSchema: {...} },
  { name: "flow.render.start", description: "Start a render job.", inputSchema: {...} },
  { name: "flow.render.status", description: "Get render job status and progress.", inputSchema: {...} },
  { name: "flow.render.wait", description: "Block until a render job completes.", inputSchema: {...} },
  { name: "flow.effects.list", description: "List all available effects.", inputSchema: {...} },
  { name: "flow.effects.describe", description: "Get an effect's parameter schema.", inputSchema: {...} },
  ...
]
```

The agent then calls tools to edit a video:

```
> tools/call { name: "flow.media.probe", args: { path: "interview.mp4" } }
{
  "duration": { "value": 1800, "rate": 30 },
  "video": { "width": 1920, "height": 1080, "codec": "h264", "fps": 30 },
  "audio": { "sample_rate": 48000, "channels": 2, "codec": "aac" }
}

> tools/call { name: "flow.timeline.apply", args: { project: "proj_01", actions: [...] } }
{ "new_action_id": "act_01HXY...", "state_hash": "abc123" }

> tools/call { name: "flow.render.start", args: { project: "proj_01", output: "out.mp4", preset: "tiktok-vertical-1080" } }
{ "job_id": "job_01HXY..." }
```

The agent subscribes to progress notifications:

```
< notifications/message { method: "flow.render.progress", params: { job_id: "job_01HXY...", percent: 42, stage: "encoding" } }
```

# Reference-level explanation

## The MCP server

`flow-server` embeds an MCP server (using a Rust MCP framework such as `mcp-rust` or `rmcp`). The server registers:

- **Tools** — one per public runtime operation.
- **Resources** — the timeline, the media list, the effect catalog, the project state.
- **Prompts** — guided workflows for common agent tasks.

## Tool catalog

| MCP tool | Maps to | Description |
|---|---|---|
| `flow.media.probe` | `runtime.probe(path)` | Get media info. |
| `flow.media.list` | `runtime.list_media(dir)` | List media in a directory. |
| `flow.project.list` | `runtime.list_projects()` | List projects. |
| `flow.project.create` | `runtime.create_project(path, spec)` | Create a project. |
| `flow.project.open` | `runtime.open_project(path)` | Open a project. |
| `flow.project.close` | `project.close()` | Close a project. |
| `flow.project.delete` | `runtime.delete_project(path)` | Delete a project. |
| `flow.timeline.get` | `project.timeline().to_otio()` | Get timeline as OTIO JSON. |
| `flow.timeline.apply` | `project.apply(action)` | Apply an action. |
| `flow.timeline.plan` | `project.plan(action)` | Validate an action without applying. |
| `flow.timeline.undo` | `project.undo()` | Undo the last action. |
| `flow.timeline.redo` | `project.redo()` | Redo. |
| `flow.timeline.history` | `project.history()` | List action history. |
| `flow.timeline.diff` | `project.diff(before, after)` | Diff two timeline states. |
| `flow.render.start` | `project.render(spec)` | Start a render job. |
| `flow.render.status` | `job.status()` | Get job status. |
| `flow.render.wait` | `job.wait()` | Block until done. |
| `flow.render.cancel` | `job.cancel()` | Cancel a job. |
| `flow.render.artifacts` | `job.artifacts()` | List output files. |
| `flow.effects.list` | `runtime.effects.list()` | List all available effects. |
| `flow.effects.describe` | `runtime.effects.describe(id)` | Get an effect's schema and docs. |
| `flow.presets.list` | `runtime.presets.list()` | List render presets. |

Every tool's `inputSchema` is auto-generated from the Rust types via `schemars`.

## Resource catalog

| MCP resource URI | Description |
|---|---|
| `flow://project/{id}/timeline` | The current timeline as OTIO JSON. |
| `flow://project/{id}/history` | The action history. |
| `flow://project/{id}/media` | The project's media references. |
| `flow://project/{id}/metadata` | The project's metadata (name, fps, resolution). |
| `flow://effects/{id}` | An effect's schema and documentation. |
| `flow://presets/{id}` | A render preset's spec. |
| `flow://projects` | The list of projects. |

Resources are subscribable; the server can push updates to clients when state changes (e.g. when another agent applies an action to the same project).

## Prompt catalog

Prompts are template-based, with parameters:

| MCP prompt | Parameters | Description |
|---|---|---|
| `flow.quickstart` | — | "You're an AI agent editing videos with Flow. Here's how to start..." |
| `flow.trim` | `{ clip_id, edge, to }` | Step-by-step guide for trimming a clip. |
| `flow.transcribe` | `{ clip_id, language }` | Guide for adding subtitles via AI transcription. |
| `flow.upscale` | `{ clip_id, scale, model }` | Guide for AI upscaling. |
| `flow.debug.render_failure` | `{ job_id }` | "A render failed. Here's how to diagnose it." |
| `flow.collaborate` | — | "You're working with another agent. Here's the protocol." |

Prompts are written by the Flow team and shipped with the server. They are versioned alongside the server.

## Error mapping

MCP errors map cleanly to Flow's typed errors:

| MCP error code | Flow error |
|---|---|
| `-32700` Parse error | Malformed JSON. |
| `-32600` Invalid request | Wrong request shape. |
| `-32601` Method not found | Unknown tool. |
| `-32602` Invalid params | Stage 1 (schema) error. |
| `-32603` Internal error | Stage 2 (semantic) or Stage 3 (precondition) error. |
| `-32000` to `-32099` Server error | Runtime errors, render failures, etc. |

The error `data` field carries a structured Flow error:

```json
{
  "code": -32602,
  "message": "Invalid params",
  "data": {
    "flow_error": "SchemaValidation",
    "details": {
      "instance_path": "/actions/0/to",
      "schema_path": "/properties/to/type",
      "message": "expected object, got number"
    }
  }
}
```

This gives agents a programmatic way to understand and react to errors.

## Streaming progress

Long-running operations (renders, AI inferences) emit MCP notifications:

```json
{
  "method": "notifications/progress",
  "params": {
    "progressToken": "job_01HXY...",
    "progress": 42,
    "total": 100,
    "message": "encoding"
  }
}
```

The `progressToken` is the `job_id`. Clients that do not subscribe to progress simply miss these notifications; the operation continues regardless.

# Alternatives

### A. Custom protocol

**Rejected.** Reinventing transport, error handling, schema negotiation. MCP exists and is supported.

### B. OpenAI function calling / Anthropic tool use directly

**Rejected.** Those are vendor-specific. MCP is the cross-vendor standard. Flow must speak MCP; vendor tool formats are client concerns.

### C. gRPC as the primary surface

**Considered.** gRPC is faster, type-safe, and streaming-native. **Rejected** because: (a) LLM clients do not speak gRPC natively, (b) gRPC is harder to debug than MCP's JSON-RPC, (c) the performance difference is irrelevant for our use case (one tool call = a few KB of JSON, latency is dominated by the action).

### D. MCP-over-HTTP only (no stdio)

**Rejected.** Some agents are subprocess-based and prefer stdio. Both should be supported.

# Drawbacks

- **MCP is a young protocol.** The spec is evolving. Flow must track MCP versions and may need to update as MCP stabilizes.
- **Tool explosion.** Flow has many tools. Listing them all in every system prompt may consume context. Mitigated by: lazy tool discovery, paginated `tools/list`, and resource-based guides.
- **Error verbosity.** Flow's typed errors are detailed. The agent may not need all the detail. Mitigated by: short `message` plus structured `data`; the agent picks what to surface.
- **Progress notifications depend on client support.** Some MCP clients ignore notifications. The agent must be prepared to poll.

# Future Possibilities

- **MCP sampling.** MCP supports the server requesting the client to invoke an LLM. A future version of Flow could use this to ask the client for clarification mid-action.
- **MCP roots.** Bind a project to a filesystem root so the agent can read media files directly.
- **MCP elicitation.** Server asks the client to confirm a destructive action. Could gate `Render` calls behind a confirmation.
- **Tool composition.** Higher-level tools like `flow.cut_filler_words(clip_id, threshold)` that compose multiple primitive actions internally.

# Unresolved Questions

1. **MCP version.** Which MCP spec version does Flow target? Recommend the latest stable.
2. **Auth.** How does `flow-server` authenticate MCP clients? Bearer tokens? mTLS? Out of scope here, but needed.
3. **Concurrency model.** Can multiple MCP clients operate on the same project simultaneously? Optimistic concurrency? Last-writer-wins?
4. **Tool versioning.** When a tool's schema changes, how do old clients cope? Embed a `version` field in tool descriptions?
5. **Local MCP.** Should `flow-cli` also expose an MCP server, or only `flow-server`? Server is cleaner; local CLI users can spawn a server.
