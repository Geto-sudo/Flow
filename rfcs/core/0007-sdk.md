# Summary

This RFC defines the **Software Development Kits** for Flow. The SDK is a set of language-specific libraries that give human developers (and programmatic agents) a typed, ergonomic API for Flow. The SDK is the third surface after the CLI (interactive use) and the MCP server (LLM use).

The v1 SDKs are:

- **`flow-script`** — Python (primary; LLM-facing).
- **`flow-rs`** — Rust (for native embedders and plugin authors).
- **`flow-ts`** — TypeScript (for browser-side use; thin wrapper over the MCP server or a future WASM core).

The SDKs are thin wrappers. They do not contain Flow logic; they call into `flow-core` (in-process) or `flow-server` (over MCP). The SDK is responsible for ergonomics, type safety, and language conventions; the runtime is responsible for correctness.

# Motivation

MCP is the right surface for LLMs. CLI is the right surface for shell scripts. But humans who write software to integrate with Flow need a language-native API:

- **Python** is the lingua franca of data science, ML, and scripting. `flow-script` is the default SDK.
- **Rust** is the language of the runtime itself and of plugin authors. `flow-rs` is the official Rust SDK.
- **TypeScript** is the language of the browser and of many automation platforms. `flow-ts` enables web UIs and Node.js automation.

Each SDK must:

1. Be **idiomatic** in its target language.
2. **Type-safe** — the JSON Schemas from the action system are the source of truth.
3. **Sync and async** — support both styles where the language allows.
4. **Multi-transport** — call in-process `flow-core`, or remote `flow-server` over MCP, with the same API.
5. **Documented** — every public function has a docstring/example.

# Goals

- Provide first-party SDKs in Python, Rust, and TypeScript.
- Support both in-process and remote transports.
- Generate types from the JSON Schemas in `schemas/`.
- Provide a fluent, fluent-scripting-style API in Python (inspired by MoviePy).
- Provide idiomatic builder patterns in Rust (the runtime's own API).
- Provide async-first APIs in TypeScript (matching the JS ecosystem's conventions).
- Document the public API of each SDK completely.

# Non Goals

- This RFC does **not** define community-contributed SDKs (Java, Go, C#, etc.). Those are encouraged but not v1.
- It does **not** redefine the runtime API; the SDKs wrap it.
- It does **not** define a CLI surface (covered by `flow-cli`, out of scope here).
- It does **not** define a web UI (out of scope; not v1).

# Guide-level explanation

### `flow-script` (Python)

```python
from flow import Video, Audio, Text, Effect, Project

# Build a timeline fluently.
clip = (
    Video("interview.mp4")
    .trim(start=5, end=15)
    .resize(width=1080, height=1920)
    .set_audio(Audio("music.mp3").volume(0.3).duck_under(clip))
    + Text("Hello").at("center").duration(2)
    + Effect("ai.upscale", scale=2)
)

# Plan (validate without applying).
plan = clip.plan()
print(plan)  # human-readable summary

# Render.
job = clip.render(output="out.mp4", preset="tiktok-vertical-1080")
job.wait()
```

Under the hood, `flow-script`:

1. Compiles the fluent expression into a list of `Action` objects.
2. Validates against the JSON Schema.
3. Submits to the runtime (in-process) or `flow-server` (via MCP).
4. Streams progress back to the caller.

### `flow-rs` (Rust)

```rust
use flow::{Project, Action, ClipAdd, ClipTrim, EffectAdd, Render};

let project = Project::open("./my-project")?;

// Apply actions imperatively.
project.apply(ClipAdd { /* ... */ })?;
project.apply(ClipTrim { /* ... */ })?;
project.apply(EffectAdd { /* ... */ })?;

// Or apply a batch.
project.apply_batch(vec![
    Box::new(ClipAdd { /* ... */ }),
    Box::new(ClipTrim { /* ... */ }),
])?;

// Render.
let job = project.render(Render::mp4("out.mp4", "tiktok-vertical-1080"))?;
job.wait()?;
```

`flow-rs` is essentially the runtime API itself, with ergonomic builder patterns. The runtime is built in Rust; the SDK is a re-export of the public API.

### `flow-ts` (TypeScript)

```typescript
import { FlowClient, Project, Action } from "@flow/sdk";

// Connect to a remote flow-server via MCP.
const client = new FlowClient({ transport: "mcp", url: "http://localhost:7700/mcp" });
const project = await client.openProject("./my-project");

// Apply actions.
await project.apply(Action.clipAdd({ /* ... */ }));
await project.apply(Action.clipTrim({ /* ... */ }));

// Render.
const job = await project.render({ output: "out.mp4", preset: "tiktok-vertical-1080" });
await job.wait();
```

`flow-ts` is async-first and uses the MCP server as its backend. (A future version could compile `flow-core` to WASM and run it in-browser; not v1.)

# Reference-level explanation

## Transport abstraction

Every SDK supports two transports:

1. **In-process** — the SDK embeds `flow-core` and calls it directly. Used by `flow-cli` and by Python scripts that have `flow-core` installed.
2. **Remote** — the SDK calls a `flow-server` over MCP. Used by web UIs, cross-machine scenarios, and by `flow-ts`.

The transport is selected at SDK initialization:

```python
# Python
client = flow.connect(in_process=True)             # embed flow-core
client = flow.connect(url="http://localhost:7700")  # call flow-server
```

```rust
// Rust
let client = flow::Client::in_process()?;
let client = flow::Client::remote("http://localhost:7700")?;
```

```typescript
// TypeScript
const client = new FlowClient({ transport: "mcp", url: "..." });  // remote only for v1
```

## Type generation

The action schemas in `schemas/` are the source of truth. Each SDK has a type generator:

```
schemas/
├── action.schema.json
├── timeline.schema.json
├── effect.schema.json
├── media.schema.json
└── ...
```

The Python generator uses `datamodel-code-generator` to produce Pydantic models. The Rust generator uses `schemars` (already in use) to produce native types. The TypeScript generator uses `json-schema-to-typescript` to produce `.d.ts` files.

The generated types are checked into the SDK repo, so SDK users do not need to run a generator.

## The fluent Python API

The fluent API is a **builder**, not a runtime. Each method returns a new immutable node. The final node compiles to a list of actions.

```python
class Video:
    def __init__(self, source: str): ...
    def trim(self, *, start: float | None = None, end: float | None = None) -> "Video": ...
    def resize(self, *, width: int, height: int) -> "Video": ...
    def set_audio(self, audio: "Audio") -> "Video": ...
    def add_effect(self, effect: str, **params) -> "Video": ...
    def __add__(self, other: "Clip") -> "Composite": ...     # composite
    def __or__(self, other: "Clip") -> "Sequence": ...      # sequence
    def __mul__(self, n: int) -> "Sequence": ...            # loop

    def plan(self) -> Plan: ...
    def render(self, *, output: str, preset: str) -> Job: ...
```

The API is inspired by MoviePy's `Clip` abstraction (RFC `moviepy.md`) but compiles to actions, not to a frame sequence. This is the key difference: MoviePy's API is fluent but its render is slow; Flow's fluent API compiles to a typed action list and renders at FFmpeg speed.

## The Rust builder pattern

```rust
pub struct ClipBuilder { /* accumulated actions */ }

impl ClipBuilder {
    pub fn new(source: MediaRef) -> Self;
    pub fn trim(self, edge: TrimEdge, to: RationalTime) -> Self;
    pub fn resize(self, w: u32, h: u32) -> Self;
    pub fn set_audio(self, audio: AudioBuilder) -> Self;
    pub fn add_effect(self, effect: impl EffectDecl, params: serde_json::Value) -> Self;
    pub fn apply_to(self, project: &Project) -> Result<()>;
}
```

Each method appends an `Action` to the builder. `apply_to` submits the batch.

## The TypeScript client

```typescript
class FlowClient {
    constructor(opts: { transport: "mcp"; url: string; token?: string });
    listProjects(): Promise<ProjectInfo[]>;
    openProject(path: string): Promise<Project>;
    createProject(path: string, spec: ProjectSpec): Promise<Project>;
}

class Project {
    readonly client: FlowClient;
    readonly id: string;
    getTimeline(): Promise<Timeline>;
    apply(actions: Action | Action[]): Promise<ActionResult>;
    undo(count?: number): Promise<void>;
    redo(count?: number): Promise<void>;
    render(spec: RenderSpec): Promise<Job>;
    close(): Promise<void>;
}

class Job {
    readonly id: string;
    status(): Promise<JobStatus>;
    progress(): AsyncIterable<Progress>;  // SSE-style stream
    wait(): Promise<JobResult>;
    cancel(): Promise<void>;
    artifacts(): Promise<Artifact[]>;
}
```

The TypeScript client is intentionally thin. All real work happens in the server.

# Alternatives

### A. One language, no SDKs

**Rejected.** The user is heterogeneous: LLMs use Python, native embedders use Rust, web UIs use TypeScript. One language would exclude a major audience.

### B. SDK as a thin CLI wrapper

**Rejected.** Subprocess-per-call is too slow and stateless. The SDK must be able to embed the runtime.

### C. Auto-generate SDKs from a single IDL

**Considered.** Use a single IDL (Protobuf, FlatBuffers, Cap'n Proto) to generate language bindings. **Deferred** because: (a) the action schemas are already JSON Schema, (b) IDL generation adds tooling complexity, (c) we can revisit if a v2 SDK needs richer types.

### D. SDK is the runtime API (no wrapper)

**Rejected.** The runtime API is a low-level, scheduler-aware, Mutex-protected internal API. The SDK is a higher-level, ergonomic surface. Different audiences, different APIs.

# Drawbacks

- **Three SDKs to maintain.** Each language has its own idioms, release cadence, and community. Mitigated by: thin wrappers, generated types, shared documentation.
- **Type generation drift.** When the schema changes, the generated types must be regenerated in all SDKs. Mitigated by: CI that regenerates types on every schema change and fails the build if the generated files are not committed.
- **In-process SDK pulls in heavy dependencies.** `flow-script` (Python) requires `flow-core` (Rust) to be installed. A pure-Python user who wants to call the remote server pays the cost of the Rust dependency for nothing. Mitigated by: separate PyPI packages: `flow-core` (with Rust extension), `flow-remote` (pure Python, RPC-only).
- **Browser support is limited.** v1 `flow-ts` is RPC-only. WASM-compiled `flow-core` is a v2 feature.

# Future Possibilities

- **WASM SDK.** Compile `flow-core` to WASM. Run Flow in the browser without a server. Heavy work; deferred.
- **More languages.** Java, Go, C#, Ruby, Elixir. Community-driven.
- **Higher-level DSLs.** A Python or TypeScript DSL that compiles to the fluent API. Or a visual builder.
- **REPL.** A Flow REPL that lets developers interactively build a timeline. Useful for exploration and debugging.

# Unresolved Questions

1. **Pydantic v1 vs v2.** Pydantic 2 is faster and has stricter typing, but the ecosystem is split. Pick one for v1.
2. **Rust API style.** Builder pattern, methods on `Project`, or a free-function API? Free functions are most flexible but least discoverable.
3. **TypeScript: MCP only or also a JSON-RPC surface?** MCP is the priority; JSON-RPC is a fallback.
4. **PyPI / crates.io / npm naming.** `flow-script`, `flow-rs`, `flow-ts`? Or `flow-py`, `flow-rust`, `flow-js`? Need to reserve names early.
5. **Async runtime in Python.** `asyncio` is the obvious choice. But many Python users prefer sync APIs. Provide both? Provide sync only and let async users wrap it?
